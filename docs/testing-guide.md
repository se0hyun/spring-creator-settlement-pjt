# 테스트 가이드

이 문서는 **수동 테스트(Swagger)·채점 시나리오**를 위해 어떤 케이스를 왜 두었는지와, 실행 절차를 한곳에 모은 것입니다. 자동 테스트(JUnit)로 옮길 때도 이 목록을 시나리오 스펙으로 쓸 수 있습니다.

---

## 1. 테스트 시나리오 작성 이유

제가 생각한 과제/ 도메인에서 중요한 포인트는 다음과 같습니다.

| 포인트 | 왜 검증해야 하는가? |
|----|-------------------|
| **집계 정확성** | 판매·환불·수수료·정산금이 **같은 규칙**으로 여러 경로(월 정산 vs 기간 집계)에서도 맞아야 함. 한 곳만 맞고 다른 곳이 틀리면 운영 사고로 이어짐. |
| **경계값** | 정산이 **월-일-시각 경계**에 민감함. `paidAt`/`canceledAt`이 달라지는 달에 걸치면 같은 거래라도 **어느 달 장부에 붙는지**가 갈림. 버그는 거의 여기서 남. |
| **carryOver(이월)** | 한 달에 **환불이 판매를 초과**하면 음수가 나오고, 그 손실을 **다음 달**에서 깎아야 함. 단순 월별 합만 보면 놓치기 쉬움. |
| **예외 및 제약** | 잘못된 금액, 역할, 중복, 잘못된 날짜 범위는 **400/404/409**로 끊어야 함. |
| **회귀 방지** | 리팩터링(DB 집계, LocalTime.MAX, Summary 초기 carry 등) 후에도 **이전에 통과했던 대표 케이스**를 다시 돌리는 용도로 쓰면 좋음. |

시나리오는 **기본 성공 경로 → 도메인 규칙 → 엣지·특이** 순으로 읽도록 구성했습니다.

## 2. 공통 전제

| 항목 | 내용 |
|------|------|
| 실행 | `./gradlew bootRun` 후 Swagger: `/api/v1/swagger-ui/index.html` |
| 데이터 | 기동 시 `DataInitializer`로 샘플 적재 (크리에이터 3명, 강의, 판매/취소 등) |

## 3. Enrollment / Cancel API — 기본&오류

| ID | 시나리오 | 목적 (왜 이 테스트를 하는가) | 절차·확인 포인트 |
|----|----------|------------------------------|-------------------|
| E-01 | 정상 구매 | 최소 성공 경로·201 | STUDENT `studentId`, 유효 `courseId`, `paidAmount` = 강의 정가, `paidAt` 유효 → **201**, 응답에 결제금·일시 일치 |
| E-02 | 결제금 ≠ 정가 | 도메인 규칙 검증 | `paidAmount`만 다른 금액으로 → **400**, `INVALID_PAYMENT_AMOUNT` |
| E-03 | 크리에이터가 구매 시도 | 역할 검증 | CREATOR 또는 MANAGER ID로 구매 → **400**, `INVALID_ENROLL_ROLE` |
| E-04 | 존재하지 않는 강의 | 리소스 검증 | 없는 `courseId` → **404** 등(ErrorCode 정의에 따름) |
| E-05 | 동일 강의 재구매 | 중복 방지 | 같은 학생·같은 강의 두 번째 구매 → **409**, `DUPLICATE_ENROLLMENT` |
| E-06 | 정상 취소 | 최소 환불 경로 | 유효 `saleRecordId`, `cancelAmount` ≤ 원결제, `canceledAt` 지정 → **200** |
| E-07 | 환불액 초과 | 검증 로직 | `cancelAmount` > `paidAmount` → **400**, `CANCEL_AMOUNT_EXCEEDED` |
| E-08 | 두 번째 취소 | idempotent 거부 | 같은 판매에 다시 취소 → **409**, `ALREADY_CANCELLED` |
| E-09 | `paidAt`/`canceledAt` 다양한 값 | 채점용 시간 조작 의도 검증 | 과거·월말·월초 등 의도적으로 바꿔 저장 가능한지 확인 (프로덕션에서는 서버 생성이 일반적이라 문서와 함께 기술) |

## 4. 월 경계·일자 — 판매/취소가 다른 달에 붙는 경우

| ID | 시나리오 | 목적 | 확인 방법 |
|----|----------|------|-----------|
| B-01 | 월말 결제 → 익월 취소 | **판매 월 ≠ 환불 월** 시 순매출이 각각 다른 달 정산에 들어가야 함 | 샘플 sale-5류: 1월 판매 분은 1월, 2월 취소 분은 2월 환불 합산에 포함되는지 월별 정산 또는 판매내역 조회로 검증 |
| B-02 | 동월 전액 환불 | 같은 달에 매출 상쇄 | 3월 판매·3월 전액 취소 시 해당 월 net 반영 |
| B-03 | 동월 부분 환불 | 부분 금액만 차감 | 원결제와 다른 환불액 시 순매출 = 판매합 − 환불합 |
| B-04 | 익월 부분 환불 | 판매 월과 환불 월 분리 | 3월 판매·4월 취소 시 환불은 **취소 발생 월** 기준 |

## 5. 월별 정산 생성 `POST /settlements/generate`

| ID | 시나리오 | 목적 | 확인 방법 |
|----|----------|------|-----------|
| G-01 | 과거 월 생성 성공 | 정상 배치 경로 | 존재하는 데이터가 있는 연월 지정 → 해당 크리에이터별 레코드 생성·금액 일치 |
| G-02 | 미래 연월 거부 | 잘못된 입력 차단 | 미래 `year/month` → **400**, `INVALID_YEAR_MONTH_VALUE` |
| G-03 | 동일 연월 재호출 | 중복 방지·멱등성 | 같은 연월 두 번 호출 → 두 번째는 생성 스킵(로그)·DB Unique 유지 |
| G-04 | 거래 없음·이월 없음 | 불필요한 행 미생성 | 해당 월 거래도 없고 이월도 없으면 레코드 생략 가능(구현 정책 확인) |

## 6. carryOver 이월 — 다월 연속 (이강사 등)

| ID | 시나리오 | 목적 | 확인 방법 |
|----|----------|------|-----------|
| C-01 | 음수 순매출 다음 달 상쇄 | 이월 규칙의 핵심 | **반드시 월 순서대로** generate (예: 1→2→…→5월). 전월 손실이 다음 달 `carryOverAmount`·`effectiveNet`에 반영되는지 |
| C-02 | 중간 달 순매출 0·정산 0 | 이월만 있는 달 | 환불만 있거나 상쇄되어 지급 0인 달도 레코드 정책·carry 연속성 확인 |
| C-03 | 마지막 달 큰 매출로 상쇄 | 큰 금액에서 이월 차감 후 정산 | 이강사 5월 등 기대 settlement·fee 확인 |

아래 표는 **월별 단건 정산** 조회 시 숫자 검증용 참고값입니다.

> `POST /settlements/generate`를 **2025년 1월 → 5월 순서**로 호출할 것 (직전 달 정산 참조).

| 월 | netSales | carryOverAmount | effectiveNet | 예상 settlementAmount |
|----|----------|-----------------|--------------|----------------------|
| 1월 | +60,000 | 0 | +60,000 | **48,000** |
| 2월 | -60,000 | 0 | -60,000 | 0 |
| 3월 | +60,000 | -60,000 | 0 | 0 |
| 4월 | -60,000 | 0 | -60,000 | 0 |
| 5월 | +120,000 | -60,000 | +60,000 | **48,000** |

## 7. 기간 집계 `GET /settlements/summary`

| ID | 시나리오 | 목적 | 확인 방법 |
|----|----------|------|-----------|
| S-01 | 정상 범위 집계 | 운영자용 합계 API 기본 | `from`≤`to`, 데이터 있는 구간 → 크리에이터별 합계·정렬 |
| S-02 | `from > to` | 입력 검증 | **400**, `INVALID_DATE_RANGE` |
| S-03 | 정렬 규칙 | 과제 후보 요구 | 정산금액 내림차순·동액 시 이름 오름차순 |
| S-04 | 월별 carry 후 합산 ≠ 단순 net×비율 | **오해하기 쉬운 검증** | 기간 내 **각 월에 이월 적용 후** fee/settlement를 더한 값. 단순히 `totalSales−totalRefunds`의 80%와 같지 않을 수 있음 (이강사 1~5월 예시 등) |
| S-05 | 직전 월 Settlement 없을 때 | 문서화된 제약 | 조회 시작 직전 달 정산이 없으면 carry 초기값이 불완전할 수 있음 → 숫자 해석 시 주의 |

## 13. 추천 실행 순서 (한 번에 다 안 돌려도 될 때)

1. **E-01 → E-06** (구매·취소 최소 성공)  
2. **E-02, E-05, E-07, E-08** (대표 에러 코드)  
3. **G-02** (미래 월)  
4. **G-03** (중복 generate)  
5. **C-01 표 순서대로 generate → M-01** (carryOver)  
6. **S-01 → S-03 → S-04** (summary)  
7. **B-01 ~ B-04** (월 경계 샘플 데이터와 교차 검증)
